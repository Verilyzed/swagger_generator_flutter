# Edge-Case Test Specs - Design

Date: 2026-06-30

## Goal

A characterization test harness for the generator: two hand-written OpenAPI
specs (3.0.x and 3.1.x) packed with rare/edge-case constructs, exercised two
ways - e2e tests that assert the exact current generated output, and inclusion
in the example so `build_runner` + `dart analyze` proves the output compiles.
The harness locks in supported behavior (regression) and surfaces gaps, which
are recorded in a prioritized list for follow-up. It does not fix the gaps.

## Decisions

- **Coverage:** broad - both regression (supported cases) and gap-finding
  (rare/unsupported cases).
- **Dialects:** both. Two specs, to exercise both `OpenApi30TypeResolver` and
  `OpenApi31TypeResolver` and each dialect's unique constructs.
- **Verification:** both - fixtures + e2e assertions, and example
  regenerate + analyze.
- **Gap policy:** document now, fix later. Characterize current behavior, record
  gaps, do not change the generator in this work.

## Approach

Hand-write the two specs as JSON. Rejected: generating from a FastAPI/Pydantic
app - it emits 3.1 only, cannot express 3.0 or deliberately malformed shapes,
and reintroduces the removed Python backend.

## Artifacts & layout

- `example/lib/specs/edge_cases_v30.json` - OpenAPI 3.0.3.
- `example/lib/specs/edge_cases_v31.json` - OpenAPI 3.1.0.
  Single source of truth; generated by `build_runner` into
  `example/lib/generated/`.
- `test/e2e/edge_cases_test.dart` - loads each spec from disk (same pattern as
  the existing `resource_scheduler` e2e test), runs `generateSources`, asserts
  the exact current output per case.
- `docs/superpowers/EDGE_CASE_GAPS.md` - the prioritized gap list; each entry a
  candidate follow-up brainstorm -> TDD cycle.

## Case inventory

To avoid duplicating every shared case in both specs, shared cases live mostly
in the 3.1 spec; the 3.0 spec focuses on 3.0-specific nullability plus a small
shared subset (so the 3.0 resolver path is still exercised on composition and
naming).

### 3.0.x spec (`edge_cases_v30.json`)

- Nullability: `nullable: true` on a field, on a `$ref` (3.0 sibling-ignored),
  and via the `allOf`-wrapper trick; `anyOf` ref-or-null.
- Shared subset on the 3.0 path: one `oneOf` multi-member, one `allOf`
  inheritance, one recursive schema, a couple of enum/name edge cases.

### 3.1.x spec (`edge_cases_v31.json`) - carries most shared cases

- 3.1 nullability: `type: ["string","null"]`; multi-type `["string","integer"]`
  and `["string","integer","null"]`; `anyOf` with `{type: null}`.
- 3.1-only: `prefixItems` tuples, `const`, boolean schemas (`true`/`false`),
  empty schema `{}`, `examples` (plural), top-level `webhooks`.
- Composition: `oneOf` multi-member, `discriminator` + `mapping`, `allOf`
  inheritance (base `$ref` + own props), nested `allOf`.
- Objects/maps/arrays: `additionalProperties` + `properties` together,
  `additionalProperties` `true`/`false`/`{}`, untyped array, array-of-arrays,
  deeply nested generics, recursive (`Tree.children`), circular `$ref` (A<->B).
- Enums: integer enum, values with spaces/hyphen/leading-digit/reserved/`$`/
  empty, single-value enum.
- Formats: `uuid`, `email`, `uri`, `byte`, `binary`, `int32`, `int64`, `float`,
  `decimal`.
- Names: kebab-case, snake_case, leading digit, reserved word, `@type`, very
  long, unicode property names.
- Parameters: header & cookie params; array/object query params with
  `style`/`explode`; `$ref` to `components/parameters`.
- Bodies: request body as array, primitive, and `oneOf`.
- Responses: `204` no content, `default`, multiple success codes, response
  `$ref`, non-JSON content (`text/plain`, `application/octet-stream`).

## How it's exercised

1. `test/e2e/edge_cases_test.dart` asserts the exact current generated output
   per case - locking in supported behavior and documenting degraded behavior
   (e.g. `oneOf` -> `dynamic`, multi-type -> `dynamic`).
2. Both specs are added to `example/lib/specs/`; `build_runner` regenerates and
   `dart analyze` on the example must stay clean.
3. A case that produces **non-compiling** code, or that **crashes generation**
   (e.g. an unbounded recursion), is carved out of the example so the build and
   analyze stay clean, kept only as an e2e fixture/assertion (asserting the
   degraded output or that generation throws), and recorded in the gap list. The
   example stays buildable and analyze-clean.

Because we cannot know up front which cases break, the specs are built
iteratively: add cases, regenerate, and move any breaker out of the example into
an e2e-only assertion plus a gap-list entry.

## Gap list

`docs/superpowers/EDGE_CASE_GAPS.md` records every case that degrades, is wrong,
or is unsupported, each with: the construct, the current behavior, the desired
behavior, and a rough priority. Each entry becomes a separate future effort.

## Out of scope

- Fixing any gap the harness reveals (separate follow-up cycles).
- Runtime round-trip against a server. The verification bar is regenerate +
  `dart analyze` plus e2e string assertions.

## Testing

- `dart test` - the new `edge_cases_test.dart` and the existing suite stay green.
- `dart analyze` clean on the package.
- `cd example && dart run build_runner build` succeeds and `dart analyze` is
  clean on the example.
